Behavioral Rules

  • Single expandedXxxId state variable per tab - only one row open at a time
  • Clicking an expanded row collapses it (toggle behavior)
  • Status dropdowns use onclick="event.stopPropagation()" to prevent expand toggle
  • Link buttons, delete buttons also stop propagation
  • Changing status triggers optimistic update (local state change + re-render, revert on API failure)
  • Expand panel left padding (3.25-3.5rem) aligns content with the row title, past the arrow and icon
  • All three tabs use the unified left-side pattern: expand arrow + type icon + label

Implementation

All admin pages include the same toast pattern. HTML: <div class="toast" id="toast"></div> before </body>.

/* CSS */
.toast { position: fixed; top: 1.25rem; left: 50%; transform: translateX(-50%);
  padding: .65rem 1.25rem; border-radius: 10px; font-size: .82rem; font-weight: 500;
  z-index: 1000; box-shadow: 0 4px 16px rgba(0,0,0,.15);
  opacity: 0; pointer-events: none; transition: opacity .3s; }
.toast.show { opacity: 1; pointer-events: auto; }
.toast.success { background: var(--color-primary); color: #fff; }
.toast.error { background: var(--color-danger); color: #fff; }

/* JS */
function toast(msg, type) {
  var t = document.getElementById('toast');
  t.textContent = msg;
  t.className = 'toast ' + (type || 'success');
  t.classList.add('show');
  setTimeout(function() { t.classList.remove('show'); }, 3000);
}
window.toast = toast;

Data Access Patterns

How pages read from and write to Supabase. All new pages use direct fetch calls. Legacy sb() and apiAction() helpers exist in some older admin pages but should not be used in new work.

Reading Data (Client-Facing & Admin)

Use the anon key for all reads. Works from both admin and client-facing pages.

var SB = 'https://ofmmwcjhdrhvxxkhcuww.supabase.co';
var SK = '...anon key...';

// Basic read
fetch(SB + '/rest/v1/contacts?slug=eq.anna-skomorovskaia&select=*', {
  headers: { 'apikey': SK, 'Authorization': 'Bearer ' + SK }
}).then(r => r.json()).then(data => { ... });

// Read with multiple filters
fetch(SB + '/rest/v1/deliverables?contact_id=eq.' + id + '&status=neq.delivered&select=*', {
  headers: { 'apikey': SK, 'Authorization': 'Bearer ' + SK }
}).then(r => r.json());
Writing Data via /api/action

All writes go through POST /api/action, which uses the service role key server-side to bypass RLS.

// Create
fetch('/api/action', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    action: 'create_record',
    table: 'deliverables',
    data: { contact_id: id, type: 'schema', title: 'Brand Schema', status: 'not_started' }
  })
}).then(r => r.json());

// Update (filters identify the row, data is the patch)
fetch('/api/action', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    action: 'update_record',
    table: 'deliverables',
    filters: { id: deliverableId },
    data: { status: 'delivered' }
  })
}).then(r => r.json());

// Delete
fetch('/api/action', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    action: 'delete_record',
    table: 'deliverables',
    filters: { id: deliverableId }
  })
}).then(r => r.json());

Actionscreate_record, read_records, update_record, bulk_update, delete_record
